# sessionsテーブル記録仕様（現状調査）

## 結論（現リポジトリ基準）

- `sessions` テーブルへ直接 `INSERT/UPDATE` するアプリコードは現状見当たらない。
- 打刻APIとcronはどちらも `logs` テーブルへの書き込みのみを行う。
- したがって、`sessions` の記録は「DB側のトリガー/バッチ（本リポジトリ外）」または「過去移行データ」に依存している可能性が高い。

## 1. 記録の起点（トリガー相当）

### A. 打刻API (`POST /api/stamp`)
- 認証済みユーザーのIN/OUT打刻を受け、`logs` に1件挿入する。
- 保存項目は `stamped_at/work_date/user_id/machine_id/decided_site_id/work_type_id/.../stamp_type` など。
- この処理内で `sessions` への書き込みはしていない。

### B. 強制OUT cron (`GET /api/cron/close-open-sessions`)
- 当日JSTで「INはあるが後続OUTがない」ログを抽出し、擬似OUTレコードを `logs` に追記する。
- 追記時に `autoGenerated=true`, `forcedSource=cron` を付与する。
- この処理でも `sessions` への書き込みはしていない。

## 2. sessionsに想定されるレコード形

読み取り側は `sessions` の生レコードをJSONとして取り出し、以下のフィールド群を吸収している。

- 基本: `id`, `date`, `start`, `end`, `durationMin`, `status`, `autoGenerated`
- 現場/業務: `siteName`, `clientName`, `workDescription`
- ユーザー: `userId`, `user`(record id), `userName`
- 機械: `machineId`, `machine`(record id), `machineName`

列名ゆれ (`machineid`, `start (JST)`, `name (from user)` など) も読み取りで吸収しているため、データ生成元が複数（Airtable由来/Neon移行由来）である前提の実装になっている。

## 3. sessionsテーブルに対する制約（DB）

マイグレーションでは `sessions` に対して以下の制約/索引のみ確認できる。

- `duration_min >= 0`
- `decision_method in ('nearest','none')`
- `status in ('open','closed')`
- `user_id`, `machine_id` 外部キー（存在時）
- `unique_key` ユニーク索引（存在時）

※ 本リポジトリ内に `CREATE TRIGGER` は存在しない。

## 4. 運用上の読み替え

現状の「sessions記録トリガー」はアプリコード視点では次のように解釈するのが妥当。

1. 打刻/cronで `logs` が増える。
2. （リポジトリ外の仕組みで）`sessions` が更新される想定。
3. レポートAPIは `sessions` を参照して勤怠集計する。

そのため、完全な仕様確定には **本番DB側のトリガー定義（`pg_trigger` / `pg_proc`）または外部ETLジョブ定義** の確認が必要。
